## 阶段执行规则

- 用户明确要求执行某一阶段时，只执行该阶段。
- 如果上一阶段缺少必要基础，只补齐最小必要内容。
- 不提前实现后续阶段功能。
- 不擅自重构无关文件。
- 不破坏已经完成的模块。
- 每次改代码前先阅读 `PROJECT_RULES.md`、`ARCHITECTURE.md`、`DATABASE_SCHEMA.md`、`DEVELOPMENT_ROADMAP.md`、`UI_GUIDE.md`。

## 阶段收尾协议

每个阶段完成后都必须自动执行 `AGENTS.md` 中的 Stage Completion Protocol。该规则适用于第 0 阶段至第 20 阶段，也适用于未来新增的开发阶段、修复阶段、UI 优化阶段、重构阶段和维护任务。

每阶段完成后必须先运行 `flutter analyze`。只有在 analyze 通过后，才允许执行 `git status`、`git add`、`git commit` 和 `git push`。如果没有任何文件变更，不创建空 commit，也不执行空 push。完成当前阶段的收尾后，不允许自动进入下一阶段。

## 第 1 阶段目标

初始化 Flutter Windows 项目，项目根目录仍必须是当前目录。推荐命令：

```bash
flutter create . --project-name jin_xiao_cun_2026 --platforms=windows
```

如果当前环境不支持 Flutter，应先说明原因，再采用等价方式初始化，但最终项目根目录必须仍然是当前目录。

## 第 20 阶段完成后的项目状态

第 0 阶段至第 20 阶段已经完成，项目具备第一版本地桌面软件的主体骨架和主要业务模块：

- 项目规则文档、架构文档、数据库规划、UI 指南和路线文档。
- Flutter Windows 项目、目录结构和基础依赖。
- 统一 theme、通用组件和桌面布局。
- SQLite + Drift 数据库、核心表、Repository 和 Service。
- 订单、客户、产品、厂家、发货、费用利润、首页看板。
- Excel 导入、Excel 导出、本地备份恢复。
- 异常提醒和数据质量检查。
- README 二次开发说明。

## 后续建议路线

后续不再按第 0-20 阶段继续自动推进，建议按维护任务或小版本迭代：

- `v0.2`：补充自动化测试、导入导出样例文件、更多异常规则验证。
- `v0.3`：完善客户/产品/厂家选择器，减少手动 UUID 输入。
- `v0.4`：完善库存入库、出库、盘点和库存流水 UI。
- `v0.5`：完善采购/加工单模块。
- `v0.6`：增强备份恢复安全性和恢复后应用重启提示。
- `v0.7`：准备移动端适配和未来同步协议。

每个小版本仍必须遵守当前分层架构、软删除、审计日志、金额整数分、字典驱动和阶段收尾协议。
